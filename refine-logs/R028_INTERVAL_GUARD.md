# R028：稀疏区间格点 Guard

## 研究问题

R027 说明 logit 量化只能提高结构一致性，不能恢复精确概率合同。R028 检验一个更弱但可证明的目标：只在稀疏区间取整端点离整数格点足够远时嵌入，否则把整个固定窗口转为擦除。

## 实现与数学合同

- 精度审计新增内部 CDF 最大漂移；支持或顺序不同时记为不可证；
- 对 `block_size=b` 报告必要条件 `max_cdf_delta < 2^-(b+1)`；
- `MicroframeConfig.cdf_uncertainty_bound` 控制编码端 guard；
- 解码端不运行 guard，避免两端 skip 决策不一致；
- 固定窗口长度不变，主动放弃窗口由认证层转换为擦除；
- 完整推导见 `docs/interval_guard_r028.md`。

## GPT-2 FP32/FP16 32-step 审计

候选顺序为 token ID，top-p=0.95。对支持一致步骤，最大内部 CDF 漂移为 `0.0041490245`。

| block size | 初始 CDF 阈值 `2^-(b+1)` | 潜在可行步骤 |
|---:|---:|---:|
| 2 | 0.125 | 20/32 |
| 4 | 0.03125 | 20/32 |
| 8 | 0.001953125 | 16/32 |
| 12 | 0.0001220703 | 10/32 |
| 16 | 0.0000076294 | 10/32 |
| 24 | 2.98e-8 | 10/32 |
| 32 | 1.16e-10 | 10/32 |

这里的“潜在可行”只检查支持一致和初始格点必要条件，不保证实际 PRF 样本具有足够 margin。

## 消息级 Guard smoke

使用 R028 轨迹的最大 CDF 漂移作为全局 bound：`epsilon=0.0041490245`。R026 的 1-byte 数据 + 8-bit tag 形成 `b=16`，初始所需 margin 为

\[
65536\times 0.0041490245=271.9105 > 0.5.
\]

因此编码端 4/4 窗口全部触发 `interval_guard_aborted`。解码端得到 4 个字节擦除，超过 2-byte RS 冗余，完整消息恢复失败。该结果是预期的安全 abstention，不是实现错误。

## 判定

`GO-THEORY / NO-GO-CAPACITY`。

Guard 的条件正确性和局部擦除机制成立，但当前 worst-case FP32/FP16 漂移下，16-bit 认证微帧没有非零可靠容量。不能声称跨精度性能提升，也不能把主动擦除解释为更低 BER 而忽略失败消息。

## 下一步 R029

优先测试 Byte-Sliced RS：每个窗口只嵌入 8-bit codeword 符号，把 `n` 从 65536 降到 256；取消逐窗口 8-bit tag，改用消息/分组级认证，错误字节先交给成熟 RS 纠正，最终由 AEAD/HMAC 验证完整消息。R028 显示 block-8 有 `16/32` 潜在可行步骤，明显高于 block-16 的 `10/32`，这是下一阶段唯一有理论依据的容量方向。
