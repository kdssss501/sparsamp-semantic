# 算法说明

## 研究问题

在不微调语言模型的前提下，能否用 SparSamp 替换指令模型的普通随机采样，
在保持模型输出分布的同时实现可靠消息嵌入，并显著提高 GPT-2 基线的语义质量？

## 输入与输出

输入：

- 能返回候选 token 概率分布的生成模型会话；
- 用户 prompt；
- 二进制载荷，按 `block_size` 分块；
- 共享秘密密钥；
- `top_p`、温度和最大 token 数。

输出：

- 生成 token IDs 与可读文本；
- 已嵌入的载荷块；
- 每步熵、采样区间、累计 KL 上界和性能指标。

## 稀疏采样

对当前消息块整数 `k`、候选数 `N=2^block_size` 和由
`HMAC-SHA256(key, prompt_hash || step)` 产生的均匀随机数 `r`：

```text
r_message = (k / N + r) mod 1
token = inverse_cdf(r_message)
[lower, upper) = token 的概率区间
t0 = ceil((lower - r) * N)
t1 = ceil((upper - r) * N)
N' = t1 - t0
k' = k - t0                 当 k + rN < N
k' = k - N - t0             其他情况
```

当 `N'=1` 时，一个消息块已经完成。解码器使用同一个模型、prompt 和密钥，
按论文的反向区间递推恢复 `k`。

概率运算在核心层转换为有理数，避免大 `block_size` 下直接用浮点数乘
`2^block_size`。模型概率本身仍受推理精度影响，因此编码和解码必须使用相同
模型 revision、量化方式、采样截断和硬件数值设置。

## 复杂度

- SparSamp 区间更新：每 token 为 `O(1)`。
- 概率分布准备：由 Provider 决定；本地模型主要成本是一次前向推理和
  top-p 候选筛选。
- 编码空间：消息状态 `O(1)`，实验轨迹 `O(T)`。
- 解码空间：单个未完成消息块需要 `O(L)` 区间记录，`L` 是该块跨越的 token 数。

## 语义质量

本方法不把消息 bit 直接映射到固定词表。消息只改变用于逆 CDF 采样的均匀
随机数；在理想精度下，输出仍服从模型的 `top_p` 分布。因此提升语义质量的
主要手段是使用更强的指令模型和更明确的 prompt，而不是牺牲分布去挑词。

## 安全和可靠性边界

- 本地完整分布后端可复现论文式分布保持论证。
- API top-20 后端只在可观测截断分布上保持分布。若 top-20 原始概率质量为
  `Z`，条件分布相对完整分布的单步偏差为 `KL(Q||P)=-log(Z)`。
- 当前载荷使用 ChaCha20-Poly1305 加密认证；采样随机数使用 HMAC-SHA256。
- 文本重新分词可能产生 Token Ambiguity。实验结果必须同时报告 token-ID
  解码和纯文本重分词解码，不能用前者掩盖后者失败。
